# maven_project_autobuild
maven project auto build and publish on remote server

该项目主要是在公司内部搭建gitlab代码服务器的情况下，配置git服务器的用户，密码， 代码库权限，Maven配置镜像nexus代理地址
同时该项目也支持拉取公网的代码

项目拉取、编译、选择待发布的微服务jar， 支持多选， 然后执行发布

# 项目发布平台使用与管理计划文档

## 一、文档概述
### 1.1 文档目的与范围
本计划文档旨在规范企业内部项目发布平台的使用流程、管理规范及运维保障机制，确保项目代码库管理、前后端发布、服务器绑定等工作高效、可控、可追溯。
适用范围覆盖平台内所有Git代码库项目及相关开发、运维、管理人员。

### 1.2 平台定位与核心价值
当前平台为**企业级项目发布与代码库管理平台**，核心价值在于：
- 统一管理Git代码库，实现代码版本可控
- 提供前后端一键发布能力，提升部署效率
- 支持服务器绑定与分支管理，保障业务稳定运行
- 内置监控与运维工具，降低运维成本

### 1.3 适用对象与角色定义
| 角色 | 职责范围 |
|------|----------|
| 管理员 | 全功能权限，负责平台配置、权限分配、审计监督 |
| 开发人员 | 负责代码提交、项目发布、分支管理 |
| 运维人员 | 负责服务器配置、环境部署、发布验证与故障处理 |
| 审计人员 | 负责操作日志审计、合规检查 |

## 三、核心流程规范
### 3.1 代码库全生命周期管理流程
1.  **新增代码库**
    - 录入git地址、项目名称、项目路径
    - 绑定项目负责人与开发团队
    - 配置基础权限与访问控制
2.  **更新代码库**
    - 定期执行「更新git代码库」操作
    - 同步最新代码版本与分支信息
    - 记录更新日志与版本变更
3.  **用户绑定与权限分配**
    - 为每个项目绑定对应开发/运维人员
    - 按角色分配查看、发布、修改权限
    - 定期回收离职/转岗人员权限
4.  **修改/删除代码库**
    - 提交修改/删除申请，经上级审批
    - 删除前备份代码库与配置信息
    - 同步更新项目清单与文档

### 3.2 项目发布流程
1.  **发布前准备**
    - 选择目标项目与发布分支（「重载分支」）
    - 绑定目标服务器与环境配置
    - 确认发布路径与构建参数
2.  **发布执行**
    - 区分「后端发布」与「前端发布」
    - 执行发布操作，监控发布日志
    - 验证服务启动状态与功能可用性
3.  **发布后验证**
    - 检查服务端口、日志输出、业务接口
    - 回滚至上一稳定版本（如发布失败）
    - 记录发布结果与问题处理情况

### 3.3 运维保障流程
- **监控与告警**：配置服务器负载、服务状态、发布成功率等监控指标，设置告警阈值
- **备份与回滚**：定期备份代码库与发布配置，支持一键回滚至上一稳定版本
- **日志审计**：记录所有发布、修改、删除操作，定期审计操作合规性

## 四、权限与安全管理
### 4.1 角色权限矩阵
| 角色 | 新增代码库 | 修改代码库 | 删除代码库 | 绑定用户 | 后端发布 | 前端发布 | 服务器绑定 | 日志审计 |
|------|------------|------------|------------|----------|----------|----------|------------|----------|
| 管理员 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 开发人员 | ❌ | ✅ | ❌ | ❌ | ✅ | ✅ | ❌ | ❌ |
| 运维人员 | ❌ | ❌ | ❌ | ❌ | ✅ | ✅ | ✅ | ❌ |
| 审计人员 | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ |

### 4.2 操作审批机制
- 生产环境发布：需运维人员验证 + 上级审批
- 删除代码库：需管理员审批 + 备份确认
- 权限变更：需提交申请，经管理员审核

### 4.3 安全基线与风险防控
- 启用Git仓库访问认证，禁止匿名访问
- 发布操作仅允许在指定IP段执行
- 定期更新服务器密码与密钥
- 建立发布失败应急预案，确保业务连续性
